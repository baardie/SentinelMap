# M4: Alerting Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Implement the alerting pipeline — geofences, watchlists, alert rules (geofence breach, watchlist match, AIS dark, speed anomaly), an alerting worker that evaluates rules against entity updates, API endpoints for CRUD + alert management, SignalR alert notifications, and a frontend alert feed with geofence rendering on the map.

**Architecture:** The AlertingWorker subscribes to Redis `entities:updated` (downstream of correlation). Each entity update is evaluated against registered `IAlertRule` implementations. Alerts are persisted to PostgreSQL and published to Redis `alerts:triggered` for SignalR broadcast. Geofence membership is tracked in Redis SETs for transition detection. AIS dark detection runs on a separate 60s timer. API endpoints expose CRUD for geofences/watchlists and alert lifecycle (acknowledge/resolve).

**Tech Stack:** .NET 9 BackgroundService, PostGIS spatial queries, Redis SETs for geofence membership, SignalR for real-time alert push, MapLibre for geofence rendering.

**Spec:** `docs/superpowers/specs/2026-03-18-sentinelmap-system-design.md` — Section 8 (Alerting & Monitoring)

**Codebase state:** M3 complete. Domain entities (Alert, Geofence, Watchlist, WatchlistEntry) already defined in M1. Enums (AlertType, AlertSeverity, AlertStatus) already exist. DbContext configurations for these tables already exist. Both AIS + ADS-B ingestion running, correlation creates entities with correct EntityType.

## Task 4: AlertingWorker + AIS Dark Detection

**Context:** The AlertingWorker subscribes to Redis `entities:updated`, deserialises the entity update, loads the full entity from the DB, evaluates all registered `IAlertRule` implementations, persists triggered alerts, and publishes them to Redis `alerts:triggered`. AIS Dark detection runs as a separate timer loop within the same worker.

**Files:**
- Create: `src/SentinelMap.Workers/Services/AlertingWorker.cs`
- Create: `src/SentinelMap.Infrastructure/Alerting/AisDarkRule.cs` (timer-based, not IAlertRule)
- Modify: `src/SentinelMap.Workers/Program.cs` (register alerting DI)

- [ ] **Step 1: Implement AlertingWorker**

```csharp
public class AlertingWorker : BackgroundService
```

Two concurrent tasks in `ExecuteAsync`:
1. **Entity update handler**: Subscribe to `entities:updated`, for each message:
   - Deserialise `EntityUpdatedMessage`
   - Load full `TrackedEntity` from DB (need position for spatial queries)
   - Evaluate all `IAlertRule` implementations in parallel
   - For each trigger: create `Alert` entity, persist, publish to `alerts:triggered`
2. **AIS Dark timer**: Every 60s, query entities where `Type=Vessel AND Status=Active AND LastSeen < now - darkTimeout`
   - Startup grace period: skip first 2 minutes after boot
   - For stale entities: set `Status=Dark`, fire AIS Dark alert
   - On reappearance (handled in entity update path): fire "Dark Period Ended" informational alert

- [ ] **Step 2: Implement AIS Dark detection logic**

Create `src/SentinelMap.Infrastructure/Alerting/AisDarkRule.cs`:

Not an `IAlertRule` — it's timer-based, not per-entity-update. It's a service that:
- Queries `IEntityRepository` for stale vessels
- Updates their status to Dark
- Creates Alert entities
- Publishes to `alerts:triggered`

Configurable dark timeout: default 15 minutes, override via `AIS_DARK_TIMEOUT_SECONDS` env var (30s for demo mode).
